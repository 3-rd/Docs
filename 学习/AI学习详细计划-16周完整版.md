## 周二｜AMP 混合精度

**目标**：理解 BF16/FP16 混合精度原理，理解为什么要用混合精度

**具体学习内容**：
- **为什么需要混合精度**：FP32 显存大速度慢；FP16 精度不够；BF16 是折中
- **BF16 vs FP16**：BF16 指数位8位（和FP32一样），更稳定，不容易 overflow
- **Loss Scaling**：为什么需要——FP16 下梯度过小变零，解决方案是放大 loss
- `torch.cuda.amp.autocast` ——自动选择 FP16/BF16
- `torch.cuda.amp.GradScaler` ——自动做 loss scaling，解决下溢
- `bfloat16` 和 `float16` 在 PyTorch 中的区别

**动手练习**：

## 周三｜DDP 分布式原理

**目标**：理解 DDP 为什么比 DP 快，梯度同步在哪里发生

**具体学习内容**：
- **DataParallel（DP）的问题**：单进程多线程，GIL 限制，通信开销大
- **DDP 原理**：多进程，每进程独立模型副本，通过 AllReduce 同步梯度
- **梯度同步时机**：反向传播后自动 AllReduce，所有 GPU 同步等待
- `torch.distributed.init_process_group` ——初始化通信组
- `torch.nn.parallel.DistributedDataParallel` ——包装模型
- `local_rank` 的概念：每个进程在本地 GPU 上操作
- **DDP 启动方式**：`torchrun` 或 `torch.distributed.launch`

**推荐资料**：PyTorch DDP 官方教程 ddp_tutorial.html

**原理理解**（不用动手，看懂即可）：
```python
# DDP 等价于以下手动 AllReduce（理解用）
for param in model.parameters():
    torch.distributed.all_reduce(param.grad, op=torch.distributed.ReduceOp.SUM)
    param.grad /= world_size  # 平均梯度
```

## 周四｜DDP 多卡实验

**目标**：真正跑通多卡训练，验证梯度同步效果

**具体学习内容**：
- 环境准备：`nvidia-smi` 确认 GPU 数量
- `torchrun` 启动多进程
- 写一个最简单的 DDP 训练脚本
- 验证：单卡 vs 双卡 loss 曲线是否一致（收敛行为相同）
- 观测：多卡训练是否真的加速了（用 time.time()）

**动手练习**（核心脚本）：
```python
# ddp_train.py
import torch, torch.nn as nn
from torch.nn.parallel import DistributedDataParallel as DDP
import torch.distributed as dist
import subprocess, os

def setup(rank, world_size):
    os.environ['MASTER_ADDR'] = 'localhost'
    os.environ['MASTER_PORT'] = '12355'
    dist.init_process_group("gloo", rank=rank, world_size=world_size)

def cleanup():
    dist.destroy_process_group()

class SimpleModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(nn.Linear(10, 32), nn.ReLU(), nn.Linear(32, 1))
    def forward(self, x):
        return self.net(x)

def train(rank, world_size):
    setup(rank, world_size)
    model = SimpleModel().to(rank)
    ddp_model = DDP(model, device_ids=[rank])
    optimizer = torch.optim.SGD(ddp_model.parameters(), lr=0.01)
    
    for epoch in range(3):
        for i in range(10):
            x = torch.randn(32, 10).to(rank)
            y = torch.randn(32, 1).to(rank)
            optimizer.zero_grad()
            output = ddp_model(x)
            loss = ((output - y) ** 2).mean()
            loss.backward()  # DDP 自动做梯度同步！
            optimizer.step()
        if rank == 0:
            print(f"Epoch {epoch}, loss={loss.item():.4f}")
    cleanup()

if __name__ == "__main__":
    subprocess.run(["torchrun", "--nproc_per_node=2", __file__])
```

## 周四｜GQA/MQA + KV Cache

**目标**：理解 KV Cache 的本质和硬件基础

**具体学习内容**：
- **MHA → MQA → GQA 的演进**：K/V head 数越来越少
  - MHA：num_kv_heads = num_attention_heads（每个 head 一组 K/V）
  - MQA：num_kv_heads = 1（所有 query 共享一组 K/V）
  - GQA：num_kv_heads < num_attention_heads（多个 query head 共享一组 K/V）
- **KV Cache 的显存计算**：Batch_size × seq_len × 2 × hidden_dim × num_kv_heads / num_attention_heads
- **为什么 GQA/MQA 能省显存**：K/V 只有少数几个 head，显存线性减少
- Llama-3 用的 GQA：num_kv_heads = 8（而 num_attention_heads = 32）

**理解题**：
```
LLaMA-3-8B:
- num_attention_heads = 32
- num_kv_heads = 8
- hidden_dim = 4096
- seq_len = 4096
- bsz = 1

问：KV Cache 在 BF16 格式下占多少 GB？
答：1 × 4096 × 2 × 4096 × 8 / 32 × 2 bytes = 16 MB（单层）
    全部32层 = 512 MB（这个数字要记住）
```

## 周二｜LoRA 原理

**目标**：理解 LoRA 为什么有效，掌握核心参数

**具体学习内容**：
- **LoRA 核心思想**：冻结预训练权重，只更新 A 和 B 两个低秩矩阵
  - W' = W + ΔW = W + BA
  - 其中 A ∼ N(0, σ²)，B = 0 初始化
  - ΔW = BA 的秩 r 远小于 W 的秩
- **为什么低秩有效**：预训练模型的本质是学一个低维流形，低秩扰动就够
- **关键参数**：
  - `r`：秩，越大越接近全量微调，但显存也越大
  - `lora_alpha`：缩放因子，实际学习率 = α/r
  - `lora_dropout`：防止过拟合
  - `target_modules`：哪些层加 LoRA（通常至少 q_proj 和 v_proj）
- **LoRA 的显存计算**：ΔW 参数量的公式

**理解题**：
```
LLaMA-7B 全量微调：
- 需要训练参数量：7B × 4 bytes（BF16）= 28 GB 显存（仅权重）

LoRA（r=8，q_proj+v_proj）：
- 参数量：2 × (4096 × 8 + 8 × 4096) = 2 × 4096 × 8 × 2 ≈ 131K params
- 显存：131K × 4 bytes ≈ 0.5 MB

问：LoRA 比全量微调省了多少倍显存？
```

## 周三｜QLoRA 原理 + PEFT 库

**目标**：理解 NF4 量化 + LoRA 的组合原理

**具体学习内容**：
- **QLoRA 核心**：NF4 量化 + LoRA + Double Quantization
  - NF4（4-bit NormalFloat）：专为神经网络设计的4位量化格式
  - 分位数量化：把权重分成多个小 blocks，每个 block 用 4-bit 表示
  - Double Quantization：对量化常数也做量化，进一步省显存
- **分位数量化的直觉**：把权重归一化到 [0,1]，每个 block 找它的分位数
- `bitsandbytes` 库使用：`BitsAndBytesConfig`
- **PEFT 库**：`from peft import LoraConfig, get_peft_model`

**动手练习**：

# W4（04-30 ~ 05-06）阶段1·中：DPO / KTO

## 周一｜DPO 原理

**目标**：理解 DPO 为什么有效，直观理解 loss 函数

**具体学习内容**：
- **传统 RLHF 的问题**：需要先训 Reward Model，再做 PPO，流程复杂且不稳定
- **DPO 的核心洞察**：DPO 把 RLHF 的 reward maximization 转化为一个简单的分类问题
- **DPO loss 直观理解**：
  - 训练数据：每个样本有（chosen 回复, rejected 回复）
  - loss = -log(σ(logit_chosen - logit_rejected))
  - 直观：让 chosen 的 logit 越大越好，rejected 的 logit 越小越好
- **DPO 不需要 Reward Model**：直接从偏好数据中学

**图解理解**：
```
传统RLHF：
  预训练模型 → SFT模型 → 训练RM → PPO优化
  （4个步骤，稳定但复杂）

DPO：
  预训练模型 → SFT模型 → 直接优化偏好
  （2个步骤，简单但需要更多偏好数据）
```

## 周二｜DPO Loss 推导 + 实践

**目标**：理解 DPO loss 公式，能用代码实现

**具体学习内容**：
- **DPO loss 公式**（定性理解即可）：
  - L = -E_{(x,y_c,y_r) ~ D}[log(σ(r_θ(x,y_c) - r_θ(x,y_r)))]
  - 其中 r_θ(x,y) = β * (log π_ref(y|x) - log π_θ(y|x))
  - 直观：chosen 的概率减去 rejected 的概率越大越好
- **β 参数**：KL 约束强度，β 越大越接近原始模型
- **偏好数据构造**（三种方式）：
  - 人工标注（成本高，质量最好）
  - SFT 模型生成 + LLM Judge 筛选（成本中等）
  - 随机生成 + 训练好的 RM 打分（成本低）

**动手练习**：
```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2-0.5B")
ref_model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2-0.5B")

def dpo_loss(model, ref_model, chosen_ids, rejected_ids, beta=0.1):
    # chosen_ids / rejected_ids: torch.Tensor [seq_len]
    
    # 计算 log ratio
    pi_chosen = model(chosen_ids.unsqueeze(0)).logits[0]
    pi_ref_chosen = ref_model(chosen_ids.unsqueeze(0)).logits[0]
    
    pi_rejected = model(rejected_ids.unsqueeze(0)).logits[0]
    pi_ref_rejected = ref_model(rejected_ids.unsqueeze(0)).logits[0]
    
    # 计算 reward
    r_chosen = beta * ((pi_chosen - pi_ref_chosen).sum())
    r_rejected = beta * ((pi_rejected - pi_ref_rejected).sum())
    
    # DPO loss
    loss = -torch.log(torch.sigmoid(r_chosen - r_rejected)).mean()
    return loss
```

## 周三｜KTO 原理 + 实践

**目标**：理解 KTO 和 DPO 的区别，知道何时用哪个

**具体学习内容**：
- **KTO（Kahneman-Tversky Optimization）**：基于行为经济学的前景理论
  - DPO 需要成对数据（chosen + rejected），KTO 只需要"好/坏"标注
  - KTO 对离群偏好数据更鲁棒（DPO 对噪声偏好敏感）
- **何时选 DPO vs KTO**：
  - 有高质量成对偏好数据 → DPO
  - 只有二元标注（好/坏）→ KTO
  - 偏好数据有噪声 → KTO
- **KTO loss 直观**：最大化"好样本的效用"同时最小化"坏样本的效用"

**理解题**：
```
场景：你想微调一个客服模型
- 数据来源1：用户对回复打分（1-5分），可以构造（高分, 低分）偏好对 → 用 DPO
- 数据来源2：只有"用户是否满意（是/否）"的标注 → 用 KTO

问：KTO 相比 DPO 在这个场景下有什么优势？
答：不需要构造偏好对，直接用二元标注训练，数据利用率更高。
```

# W5（05-07 ~ 05-13）阶段1·下：GRPO + RLHF 完整流程

## 周一｜GRPO 原理

**目标**：理解 GRPO 为什么是 DeepSeek 的核心技术

**具体学习内容**：
- **GRPO 核心思想**：Group Relative + 不用 Critic Network
  - PPO 需要一个 Critic（价值网络）来估计 V(s)
  - GRPO 用 group 内相对奖励代替 Critic
- **GRPO 采样**：对同一个 prompt 生成 G 个回复，计算每个回复的 reward
- **GRPO loss**：基于 group 内相对优势进行策略更新
  - advantage_i = r_i - mean(r_1...r_G)
  - policy loss = -min(risk_ratio × advantage, clip × advantage)
- **为什么 GRPO 省显存**：不需要单独训练一个 Critic，显存占用大幅减少

**核心对比**：
```
PPO（需要 Critic）：
  - 4 个网络：Policy / Critic / Reference / Reward
  - 显存占用大（Critic 通常和 Policy 一样大）
  - 训练不稳定（Critic 和 Policy 同时更新）

GRPO（不需要 Critic）：
  - 3 个网络：Policy / Reference / Reward
  - 显存占用小（没有 Critic）
  - 训练更稳定（reward 直接来自 environment）
```

## 周二｜PPO 基础

**目标**：理解 PPO 的四步骤，理解 GAE

**具体学习内容**：
- **PPO 四步骤**：
  1. Rollout：采集一批 (s, a, r, s') 数据
  2. Advantage Estimation：用 GAE 估计 advantage
  3. PPO Clip：限制策略更新幅度
  4. Gradient Update：多轮 SGD
- **GAE（Generalized Advantage Estimation）**：
  - advantage ≈ r_t + γ·r_{t+1} + ... + γ^{T-t}·V(s_T) - V(s_t)
  - λ 参数控制偏差-方差权衡（λ 越大越无偏，越小越低方差）
- **PPO Clip 机制**：

## 周二｜PagedAttention 原理

**目标**：深入理解 PagedAttention

**具体内容**：
- **传统 KV Cache 的问题**：PagedAttention 之前，KV Cache 必须是连续内存，实际使用率只有 20-40%
- **PagedAttention 核心思想**：把 KV Cache 分成固定大小的 block（类似操作系统的分页内存）
- **Block Table**：虚拟地址 → 物理地址的映射表
- **显存利用率**：PagedAttention 后，显存利用率可以达到 90%+

**理解题**：
```
传统方式（连续内存）：
- 假设 seq_len=4096，预分配 4096 个位置的连续内存
- 如果实际只用了 1000 个token，浪费了 75%

PagedAttention：
- 把 KV Cache 分成 16 个 block，每个 block 64 个 token
- 按需分配，只分配实际使用的 block
- 利用率：实际使用量 / 分配量 ≈ 90%
```

## 周三｜Continuous Batching 机制

**目标**：理解 vLLM 的请求调度

**具体内容**：
- **Static Batching**：所有请求组成一个固定 batch，一起处理完
  - 缺点：短请求等待长请求，GPU 利用率低
- **Continuous Batching（Iteration-level Batching）**：
  - 每个请求独立迭代，新请求随时加入 batch
  - 某个请求完成，立即释放，其 slot 被新请求占据
  - GPU 利用率大幅提升
- **调度策略**：Prefill 请求优先还是 Decode 请求优先
