## 操作流程

### 阶段 1：建框架

1. 参考 16 周学习计划，确认本周（Wn）的学习主题
2. 为本周每天创建目录 `WnD1/`, `WnD2/`, ...
3. 为每天创建 ipynb 框架（含十大部分标题占位）
4. 生成分 section 的 md 骨架（知识点大纲，不填详细内容）

### 阶段 2：讨论（核心）

1. 付晨阳先看 `sections/` 下的 md 基础内容
2. 带着问题来讨论
3. 讨论完成后，记录讨论中延伸的补充知识和问答

### 阶段 3：刷新

1. 用 `build.py` 把最终讨论内容刷入 ipynb 对应 section
2. 每 section 完成即 commit，不等整周完成
3. 更新 `manifest.md` 跟踪进度

## build.py 使用规范

**目录结构**：
```
W1D{n}/
├── W1D{n}.ipynb       ← 自动生成
├── build.py            ← 构建脚本
└── sections/          ← 每个子章节一个 .md 文件
```

**sections/ 下 md 格式**：
- markdown 解释 + ` ```python ` 代码块交替
- build.py 按 ` ```python ` 分隔符自动拆分成 md cell 和 code cell

**关键 bug**：正则表达式必须用 `\n``` `（closing ``` 前必须有换行），否则会吞掉后续内容。

**工作流**：
1. 编辑 `sections/1-N-xxx.md`
2. 运行 `python3 build.py` → 自动生成 W1D{n}.ipynb
3. 不直接编辑 .ipynb 文件

## 质量保证机制

1. **大纲先对齐**：生成完日框架后先让付晨阳审核，确认后再填详细内容
2. **频繁可见**：每个 section 讨论完立即 commit，付晨阳随时能看
3. **文件即记忆**：manifest + md 文件保证不依赖 context
4. **讨论驱动**：付晨阳看 md 后提问，AI 解答，不在基础知识讲解上浪费时间

---

## 操作原则

1. 每次操作前说明要做什么，等付晨阳说"好"再执行
2. 每 section 讨论完立即 commit，不等整周完成
3. study-notes 目录只生成/修改，不删除
